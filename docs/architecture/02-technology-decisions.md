## ADR-001: React + Vite للواجهة الأمامية

### القرار
استخدام React 19 مع Vite كأداة بناء.

### السبب
- React هو الأكثر انتشاراً ولديه أكبر مجتمع مطورين
- Vite سريع في التطوير (HMR فوري) وفي البناء للإنتاج
- دعم ممتاز لـ TypeScript
- مكتبة shadcn/ui المتوافقة مع React توفر مكونات UI جاهزة
- TanStack Query (React Query) لإدارة البيانات من الـ API

### البدائل المستبعدة
| البديل | سبب الاستبعاد |
|--------|---------------|
| Next.js | تعقيد إضافي غير مطلوب (SSR ليس ضرورياً لهذا النوع من التطبيقات) |
| Vue.js | مجتمع أصغر، مكتبات UI أقل نضجاً |
| Angular | منحنى تعلم أعلى، حجم أكبر |
| Svelte | مجتمع صغير نسبياً، أقل دعم مكتبات |

## ADR-002: Express 5 + Node.js للـ API

### القرار
استخدام Express 5 مع Node.js كخادم API.

### السبب
- Express 5 ناضج ومستقر ومعتمد على نطاق واسع
- بيئة Node.js تتيح TypeScript مشترك بين الواجهة والخادم
- أداء ممتاز للتطبيقات ذات الـ I/O المكثف
- متوافق مع بيئة Replit

### البدائل المستبعدة
| البديل | سبب الاستبعاد |
|--------|---------------|
| Fastify | أسرع لكن بيئة Express أكثر نضجاً ودعماً |
| NestJS | تعقيد إضافي غير ضروري لهذا المشروع |
| Python/FastAPI | يتطلب بيئة منفصلة عن الواجهة |
| Go/Gin | أداء أعلى لكن لا يشارك TypeScript مع الواجهة |

## ADR-003: PostgreSQL + Drizzle ORM لقاعدة البيانات

### القرار
استخدام PostgreSQL كقاعدة بيانات رئيسية مع Drizzle ORM.

### السبب
- PostgreSQL: أقوى قاعدة بيانات مفتوحة المصدر، تدعم JSON وfull-text search
- Drizzle ORM: خفيف، type-safe بالكامل مع TypeScript، أداء ممتاز
- متوفر مجاناً في بيئة Replit
- يدعم rollback تلقائياً

### البدائل المستبعدة
| البديل | سبب الاستبعاد |
|--------|---------------|
| MongoDB | لا يدعم العلاقات بكفاءة (مطلوب للفرق والصلاحيات) |
| MySQL | PostgreSQL أقوى في JSON والميزات المتقدمة |
| Prisma ORM | أبطأ من Drizzle وحجم أكبر |
| TypeORM | أقل type-safety من Drizzle |

## ADR-004: OpenAPI + Orval للعقد بين الواجهة والخادم

### القرار
استخدام OpenAPI 3.1 كمصدر وحيد للحقيقة (single source of truth)، مع Orval لتوليد hooks React Query و Zod schemas.

### السبب
- يضمن تطابق تام بين الواجهة والخادم
- يوفر وقت كبير بتوليد الكود تلقائياً
- يقلل الأخطاء البشرية
- يوثق الـ API تلقائياً

### البدائل المستبعدة
| البديل | سبب الاستبعاد |
|--------|---------------|
| كتابة API يدوياً | خطر عدم التطابق بين الواجهة والخادم |
| tRPC | لا يدعم OpenAPI spec، صعب التوثيق |
| GraphQL | تعقيد إضافي غير مبرر لهذا المشروع |

## ADR-005: نظام الوكلاء المتعدد بمحرك مركزي

### القرار
استخدام معمارية وكلاء متعددين (Multi-Agent) بمحرك تنفيذ مركزي بدل وكيل واحد يفعل كل شيء.

### السبب
- كل وكيل متخصص = نتائج أفضل
- أسهل في التحكم بالتوكن (حد لكل وكيل)
- أسهل في تحديد الأخطاء (أي وكيل فشل؟)
- يمكن استخدام نماذج مختلفة القوة لكل وكيل

### الوكلاء المخططون
| الوكيل | المهمة | نموذج AI مقترح |
|--------|--------|---------------|
| Code Generator | توليد الكود | GPT-4 / Claude 3.5 Sonnet (قوي) |
| Code Reviewer | مراجعة الكود | GPT-4 / Claude 3.5 Sonnet (قوي) |
| Fix Agent | إصلاح الأخطاء | GPT-4 / Claude 3.5 Sonnet (قوي) |
| File Manager | إدارة الملفات | GPT-3.5 / Haiku (خفيف) |

### البدائل المستبعدة
| البديل | سبب الاستبعاد |
|--------|---------------|
| وكيل واحد شامل | يستهلك توكن كثيرة، صعب التحكم به |
| LangChain Agents | تعقيد إضافي، overhead في التوكن |

## ADR-006: نظام ترجمة ثنائي اللغة (i18n)

### القرار
استخدام مكتبة `react-i18next` لدعم العربية والإنجليزية مع تبديل RTL/LTR.

### السبب
- أكثر مكتبة i18n نضجاً في بيئة React
- تدعم تحميل الترجمات بشكل كسول (lazy loading)
- تدعم interpolation و pluralization
- مجتمع كبير ووثائق شاملة

### البدائل المستبعدة
| البديل | سبب الاستبعاد |
|--------|---------------|
| next-intl | مرتبط بـ Next.js فقط |
| FormatJS/react-intl | أكثر تعقيداً في الإعداد |
| ترجمة يدوية | غير قابل للصيانة مع نمو المشروع |

## ADR-008: Replit Auth للمصادقة

### القرار
استخدام Replit Auth (OpenID Connect + PKCE) للمصادقة في المرحلة الأولى.

### السبب
- جاهز للاستخدام بدون إعداد معقد
- آمن (PKCE flow)
- متكامل مع بيئة Replit
- لا يحتاج إدارة كلمات مرور

### ملاحظة
في مراحل لاحقة يمكن إضافة مصادقة مستقلة (email/password) عبر مكتبة مثل Lucia Auth أو Auth.js.

## ADR-009: Redis + BullMQ لقائمة انتظار المهام (Task Queue)

### القرار
استخدام Redis كطبقة وسيطة لقائمة انتظار المهام، مع مكتبة BullMQ لإدارة الـ jobs والـ workers.

### السبب
- Redis: سريع جداً (in-memory)، يدعم pub/sub للإشعارات الفورية
- BullMQ: مكتبة ناضجة مبنية على Redis، تدعم: إعادة المحاولة (retry)، الأولويات (priorities)، التأخير (delayed jobs)، المراقبة (monitoring)
- يتيح فصل محرك التنفيذ عن الوكلاء: المحرك يضيف المهام، والوكلاء يستهلكونها
- يدعم التنفيذ المتوازي (عدة وكلاء يعملون في نفس الوقت على مشاريع مختلفة)
- يدعم rate limiting لكل وكيل (لضبط حد التوكن)
- يوفر dashboard مراقبة جاهز (Bull Board)

### كيف يعمل مع محرك الوكلاء
```
محرك التنفيذ ──► يضيف مهمة إلى Redis Queue
                        │
            BullMQ Worker (Code Generator) ──► يأخذ المهمة
                        │
            ينفذها ──► يُرجع النتيجة إلى Redis
                        │
            محرك التنفيذ ──► يأخذ النتيجة ──► المهمة التالية
```

### أنواع Queues المطلوبة
| Queue | الغرض |
|-------|-------|
| `build:codegen` | مهام توليد الكود |
| `build:review` | مهام مراجعة الكود |
| `build:fix` | مهام إصلاح الأخطاء |
| `build:files` | مهام إدارة الملفات |
| `notifications` | إشعارات المستخدمين |

### البدائل المستبعدة
| البديل | سبب الاستبعاد |
|--------|---------------|
| PostgreSQL كـ queue (pg-boss) | أبطأ من Redis، يضغط على قاعدة البيانات الرئيسية |
| RabbitMQ | تعقيد إضافي في الإعداد والصيانة |
| AWS SQS | خارجي ومُكلف، يتطلب حساب AWS |
| Queue في الذاكرة فقط | يضيع المهام عند إعادة التشغيل |
