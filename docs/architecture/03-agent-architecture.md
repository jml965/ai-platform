# معمارية نظام الوكلاء — Agent Architecture

---

## نظرة عامة

نظام الوكلاء هو قلب المنصة. يتكون من محرك تنفيذ مركزي يدير 4 أنواع من الوكلاء، كل منهم متخصص في مهمة واحدة.

---

## مكونات النظام

### 1. محرك التنفيذ (Execution Engine)

المسؤول عن:
- استقبال طلبات البناء من الـ API
- تقسيم الطلب الكبير إلى مهام صغيرة
- وضع المهام في قائمة انتظار (Task Queue)
- توزيع المهام على الوكلاء بالترتيب الصحيح
- تطبيق دستور الوكلاء (Agent Constitution)
- تسجيل كل عملية في سجل التنفيذ

```
طلب البناء ──► محرك التنفيذ ──► قائمة الانتظار ──► الوكيل المناسب
                    │
                    ├── يتحقق من حد التوكن
                    ├── يختار الوكيل المناسب
                    └── يسجل في سجل التنفيذ
```

### 2. قائمة الانتظار (Task Queue — Redis + BullMQ)

- مبنية على Redis كطبقة تخزين مؤقتة (in-memory) مع BullMQ لإدارة الـ jobs والـ workers
- 5 queues منفصلة: `build:codegen`، `build:review`، `build:fix`، `build:files`، `notifications`
- كل مهمة لها حالة: `pending` → `in_progress` → `completed` / `failed`
- المهام تُنفَّذ بالتسلسل داخل المشروع الواحد
- المهام من مشاريع مختلفة يمكن أن تعمل بالتوازي
- BullMQ يوفر: إعادة المحاولة التلقائية، الأولويات، rate limiting لكل وكيل

## دستور الوكلاء (Agent Constitution)

### القواعد العامة

| # | القاعدة | الشرح |
|---|---------|-------|
| 1 | الوكيل يعمل ضمن مرحلته فقط | لا يمكن لوكيل توليد أن يراجع |
| 2 | لا تواصل مباشر بين الوكلاء | كل التواصل عبر محرك التنفيذ |
| 3 | حد توكن لكل مهمة | يُوقَف الوكيل عند تجاوز الحد |
| 4 | ملفات المشروع فقط | لا يمكن تعديل ملفات النظام |
| 5 | تعديل جزئي فقط | عند الإصلاح، الجزء المتأثر فقط |
| 6 | تسجيل كل عملية | النتيجة + الملفات + التوكن + المدة |

### صلاحيات الملفات

| العملية | Code Gen | Code Review | Fix Agent | File Manager |
|---------|----------|-------------|-----------|-------------|
| إنشاء ملف | نعم | لا | لا | نعم |
| تعديل ملف | لا | لا | نعم (الجزء المتأثر) | نعم |
| حذف ملف | لا | لا | لا | نعم (بموافقة المحرك) |
| قراءة ملف | نعم (الملف الحالي) | نعم (الملف المراد مراجعته) | نعم (الملف الخاطئ) | نعم |

## تدفق مهمة البناء التفصيلي

```
1. المستخدم يرسل: "أريد صفحة تسجيل دخول بسيطة"

2. محرك التنفيذ:
   ├── يحلل الطلب
   ├── يقسمه إلى مهام:
   │   ├── مهمة 1: توليد login.html
   │   ├── مهمة 2: توليد login.css
   │   └── مهمة 3: توليد login.js
   └── يضع المهام في القائمة

3. مهمة 1 ──► Code Generator:
   ├── يستقبل: "اكتب HTML لصفحة تسجيل دخول"
   ├── يُرسل للـ AI API
   ├── يستقبل الكود
   ├── يُسجل التوكن المستخدم
   └── يُسلم الكود لمحرك التنفيذ

4. محرك التنفيذ ──► Code Reviewer:
   ├── يستقبل: الكود المولَّد
   ├── يفحص: هل هو HTML صالح؟ هل فيه أخطاء؟
   ├── إذا نجح ──► حفظ الملف
   └── إذا فشل ──► Fix Agent

5. Fix Agent (إذا لزم):
   ├── يستقبل: الكود + رسالة الخطأ
   ├── يُصلح الجزء المتأثر فقط
   └── يُعيد إلى Code Reviewer (حد 3 محاولات)

6. حفظ الملف ──► تحديث المعاينة ──► المستخدم يرى النتيجة
```

## حالات المهمة

```
pending ──► in_progress ──► completed
                │
                └──► failed ──► retry (حد 3) ──► permanently_failed
```

## Prompt Templates (أمثلة مختصرة)

### Code Generator Prompt
```
أنت مطور ويب. اكتب {fileType} لـ:
{userDescription}

القواعد:
- كود نظيف ومقروء
- استخدم أسماء دلالية
- لا تضف تعليقات غير ضرورية
- أرجع الكود فقط بدون شرح
```

### Code Reviewer Prompt
```
راجع هذا الكود:
{generatedCode}

افحص:
1. هل الكود صالح syntactically؟
2. هل فيه أخطاء منطقية؟
3. هل يحقق المطلوب: {requirements}؟

أرجع: passed أو failed مع قائمة المشاكل
```

### Fix Agent Prompt
```
هذا الكود فيه خطأ:
{errorCode}

رسالة الخطأ:
{errorMessage}

أصلح الجزء المتأثر فقط.
لا تعيد كتابة الملف بالكامل.
أرجع الكود المُصلح فقط.
```
